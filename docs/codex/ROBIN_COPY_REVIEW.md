# Robin Copy Review

## Verdict
YELLOW

## One-Sentence Read
EasyLife sounds calmer and more concrete than before, but it still slips into product-builder language and broad suite claims where the user needs a plain daily action.

## Mission Voice Fit
The strongest language fits the mission when it uses concrete personal-workspace nouns: tasks, notes, calendar planning, workouts, settings, workspace, daily plan. That voice supports a connected, professional personal suite. The weaker language still frames EasyLife as a "command center" or a demo/proof surface, which makes the product feel like a pitch or build artifact instead of a trusted place to decide what to do today.

## Delicate Wording Risks
- "command center" appears in customer-facing meta copy and conflicts with the operating mode's ban on generic command-center framing.
- "Open EasyLife" can read like a public marketing CTA when used on protected or product-adjacent surfaces; it does not tell the user what they will do next.
- "demo", "proof", and "sample" appear in source smoke hits. Some may be internal identifiers, but any visible use should be replaced with "example day", "workspace", "preview", or the actual object shown.
- "EasyLife keeps tasks, notes, calendar planning, workouts, projects, and pipeline work in one command center" is a feature inventory and a broad product claim; it over-explains before the user knows the daily outcome.
- Repeated product-page phrasing risks making core modules feel like brochures rather than working tools.
- Any phrase like "polish", "handoff", or "workflow" should be treated as suspect unless the reader, action, and result are concrete.

## Beautiful Language Opportunities
- Replace suite claims with a smaller daily promise: help the user see today, capture what comes up, and return to the right list, note, block, or workout.
- Give route-specific CTAs practical verbs: "Review today", "Add a task", "Capture a note", "Plan the next block", "Log workout".
- Use "workspace" carefully as a home base, not as a vague value prop.
- Let each module name its primary object first: task list, note library, calendar block, workout log, settings.
- Shorten first-screen helper copy so it supports one next move instead of describing the whole system.
- Use "example" or "preview" only when a surface is truly mock/static; otherwise name the actual user action.

## Priority Rewrite
Fix the remaining "command center" and demo/proof language in visible public and app copy. The next copy task should replace broad builder phrases with concrete daily-workspace wording, starting with meta/page descriptions and any visible CTA or preview label that still sounds like a sales artifact rather than something a user can act on.

## Voice Rules
- Lead with what the user can do now, not what the product contains.
- Prefer concrete nouns: task list, note, calendar block, workout log, project brief, settings, workspace.
- Avoid "command center", "personal operating system", "premium", "high-tech", "demo", "proof", "handoff", and "polish" in visible copy.
- Keep first-screen labels short; move explanations below the fold or behind existing detail views.
- Separate marketing copy from app copy: public pages can explain, working app screens should direct action.
- Do not invent intelligence, automation, backend, or AI claims unless the visible feature actually exists.
- Use calm verbs: review, add, capture, plan, choose, log, adjust, continue.
- Every visible sentence should answer: who is this for, what should they do, and what do they get?
