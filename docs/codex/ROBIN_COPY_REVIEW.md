# Robin Copy Review

## Verdict
YELLOW

## One-Sentence Read
EasyLife is close to a calm personal workspace voice, but scattered builder words and first-screen over-explaining still make parts of it sound like an internal product review instead of a daily assistant.

## Mission Voice Fit
The language mostly fits the mission when it uses concrete daily nouns: workspace, daily plan, tasks, notes, calendar, workouts, settings, project brief. The weaker spots are where EasyLife calls itself a "command center," uses process words like "handoff," "proof," "polish," "demo," or asks the user to parse too many helpful explanations before the next action is clear. The product should sound like a trusted daily tool, not a pitch deck or build log.

## Delicate Wording Risks
- "command center" in public metadata and app copy feels generic SaaS and too forceful for a calm personal assistant.
- "Quick blockers and handoffs." sounds like team operations jargon, not personal calendar language.
- "Start with" can read like builder instruction copy unless the object and user outcome are concrete.
- "proof," "polish," and "demo" should stay out of customer-facing UI unless they are clearly internal identifiers.
- "sample data" or demo-mode copy risks making the product feel fake when shown in visible examples.
- "workflow" is too vague as a standalone value noun; prefer task list, calendar block, project brief, note, or workout log.
- Settings still appears to over-explain too much above the fold; the copy burden makes the control-center job feel less immediate.
- EasyProjects sample/timeline language still risks sounding like shipping work rather than a user's project planning.

## Beautiful Language Opportunities
- Make HQ feel like a morning opening: one clear action, today context, and quiet module status.
- Replace process language with daily-life objects: "project brief," "calendar prep," "next task," "recent note," "today's plan."
- Let Settings lead with one concrete control group instead of explanatory copy.
- Make EasyProjects examples sound like real personal projects, not product-development tasks.
- Use shorter labels in app chrome so the working surface feels faster and less announced.
- Public pages can become warmer by showing useful module moments rather than repeating feature-category language.

## Priority Rewrite
The single most important wording problem is remaining internal/process language in visible product surfaces, especially "command center," "handoff," "proof," "polish," "demo," and "sample data." Nami should replace these one phrase at a time with concrete user outcomes and daily objects, without adding longer explanations or new claims.

## Voice Rules
- Prefer concrete daily nouns over product-process nouns.
- Keep first-screen copy short: one job, one next action, one context cue.
- Do not use "command center," "personal operating system," "premium," or "high-tech" in visible user-facing copy.
- Avoid "proof," "polish," "handoff," "workflow," "demo," and "sample data" unless clearly internal and not rendered to users.
- Public pages may explain the product; signed-in app screens should use direct task labels.
- Empty states should answer: what is this for, what should I do, and what do I get?
- Settings copy should name the control and outcome, not narrate the system.
- Example content should sound like real life: plan week, finish brief, prep calendar, review note, log workout.
